# Glue Coding 方法论

## 定义

**Glue Coding** 是一种新的软件构建方式，核心理念是：

> **几乎完全复用成熟的开源组件，用最少量的"胶水代码"将它们组合成完整系统。**

强调"连接"而不是"创造"，在 AI 时代尤其高效。

---

## 背景

传统软件工程要求开发者：
- 设计架构
- 自己编写逻辑
- 手动处理各种细节
- 反复重复造轮子

这导致：开发成本高、周期长、成功率低。

当前生态已根本改变：
- GitHub 上有数千个成熟开源库
- 框架覆盖各种场景（Web、AI、分布式、模型推理…）
- GPT / AI 可以帮助搜索、分析、组合这些项目

因此，"胶水编程"成为新范式。

---

## 核心原则

### 1. 能不写就不写，必须写就少写
任何有成熟实现的功能都不应重复发明。

### 2. 能复制就复制
直接复制使用社区验证过的代码是正常工程实践，不是偷懒。

### 3. 站在巨人肩膀上
利用现有框架，而不是自己写另一个"更好的轮子"。

### 4. 不修改上游仓库代码
所有开源库尽可能保持不变，作为黑盒使用。

### 5. 自定义代码越少越好
你写的代码只负责：
- 组合
- 调用
- 封装
- 适配

这就是 **胶水层**。

---

## 标准流程

### 1. 明确需求
将系统功能分解为独立的需求点。

### 2. 用 AI 分解需求
让 AI 将需求细化为可复用的模块、能力点和子任务。

### 3. 搜索现有开源实现
使用 AI 搜索能力：
- 搜索对应每个子需求的 GitHub 仓库
- 检查是否有可复用组件
- 比较质量、实现方式、许可证

### 4. 下载并组织仓库
拉取选中的仓库到本地并组织。

### 5. 按架构组织
将仓库放入项目结构：
```
/services
/libs
/third_party
/glue
```

强调：**开源仓库是第三方依赖，不得修改。**

### 6. 编写胶水层代码
胶水代码的职责：
- 封装接口
- 统一输入输出
- 连接不同组件
- 实现最小业务逻辑

最终系统由多个成熟模块组装而成。

---

## Glue Coding 的价值

| 方面 | 价值 |
|------|------|
| 成功率 | 极高，使用社区验证的成熟代码 |
| 开发速度 | 极快，大量功能可直接复用 |
| 成本 | 大幅降低时间、维护、学习成本 |
| 稳定性 | 更高，依赖成熟框架 |
| 扩展性 | 容易，通过替换组件升级能力 |
| AI 兼容 | 高，AI 可辅助搜索、分解、集成 |

---

## Glue Coding vs 传统开发

| 项目 | 传统开发 | Glue Coding |
|------|----------|-------------|
| 功能实现方式 | 自己编写 | 复用开源 |
| 工作量 | 大 | 大幅减少 |
| 成功率 | 不确定 | 高 |
| 速度 | 慢 | 极快 |
| 出错率 | 容易踩坑 | 使用成熟方案 |
| 关注点 | "发明轮子" | "组合轮子" |

---

## 典型应用场景

- 快速原型开发
- 小团队构建大系统
- AI 应用 / 模型推理平台
- 数据处理管线
- 内部工具开发
- 系统集成

---

## 未来趋势

随着 AI 能力不断增强，未来开发者将不再需要自己编写大量代码，而是：

- 找轮子
- 组合轮子
- 智能连接组件
- 用极低成本构建复杂系统

**Glue Coding 将成为软件生产力的新标准。**
